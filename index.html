opengl图片和纹理

发表于 2020-04-16 | 分类于 opengl |

一、图片和纹理

图片的读取在移动设备中，左上角是 （0，0）点，所以图片的读取是从左上角开始读取的

纹理坐标系是

图片的读入是从左上角开始读入，放入到纹理坐标中是从纹理坐标的左下角开始放入，所以默认情况下是上下颠倒的，即从纹理坐标下面读入的是图片上面的内容

如何正确的从纹理坐标中按照图片正确方向读入内容呢

从纹理坐标的左上角开始读入，左上—->右上—>右下—>左下 的顺序，则是从下到上读入图片的内容，

然后放入到顶点坐标系中，也是从左下开始放入，此时图片就是正的

二、视口的作用

glViewport(x,y,width,height) 决定了在GLSurfaceView大小内我们选取的绘图区域，默认全部使用

坐标系 左下 —————> 右上

正交投影和透视投影用于顶点坐标系 顶点坐标系 顶点着色器变换

视口大小(即绘图区域)

使得纹理内容能够在绘图区域约束下，不变形的绘制(实际是纹理被按照更小的绘图区域绘制使用)

即改变纹理的填充区域，默认情况下纹理是填充满视口区域的，但是在正交变换或者透视变换作用下，纹理的填充区域被按照纹理的宽高比例缩小了(未绘制的部分是黑色区域)

三、加载过程的理解

图片—–>纹理坐标—-> 正交顶点坐标系—>正交或者透视投影 —-> 有效视口绘制

1是纹理内容

2是视口总的大小

阴影区域是通过正交变换，实际纹理的使用大小

Thread生成后的状态

NEW：至今尚未启动的线程处于这种状态。
RUNNABLE：正在 Java 虚拟机中执行的线程处于这种状态。
BLOCKED：受阻塞并等待某个监视器锁的线程处于这种状态。—等待synchronized锁 或者 主动调用 object.wait()

TIMED_WAITING：等待另一个线程来执行取决于指定等待时间的操作的线程处于这种状态。

非静态Join方法

Thread a中调用Thread b的 b.join() 方法，表现是线程a一直阻塞等到b执行完成后在执行b.join()之后的内容

1、 在a线程中调用b线程的 b.join(), 当b线程是alive情况（即线程b没有中断或者run方法没有执行完成），则一直阻塞a线程

2、 run方法为线程体，run方法运行结束则线程终止，而cpu再运行其他线程

3、注意b线程中，只有run方法的线程id是b线程的id，而执行b线程的其他方法时候的线程id还是 调用者的线程id

4、所以说b线程一直被阻塞，阻塞就会释放a线程的锁，所以说join方法执行后，a线程不再持有锁，当b线程结束时候就会结束阻塞

#1、内存泄漏原因
常见的内存泄漏原因是由于 引用导致，即对象引用时导致对象不可以释放的根本原因，对象不能释放，所以导致内存泄漏。

#2、内存泄漏分析工具
分析工具采用android device monitor 、MAT、GIMP工具来分析内存
MAT地址
http://eclipse.org/mat/downloads.php
GIMP下载地址
https://www.gimp.org/
首先要导出hprof文件，并利用命令行进行hprof-conv转换，具体操作，百度搜索，注意一点，在进行点击app页面通过android device monitor来进行获取hprof文件的时候，需要退出所监听内存的页面，并点击一下GC，再进行dump hprof文件。

#3、具体内存泄漏分析方法
查找内存泄漏对象的方法

参考 https://blog.csdn.net/yincheng886337/article/details/50524890
1、通过日志OOM现象，切换到同一个页面，不断进行重复切换过程中，发现切换到同一个页面，内存有增加现象，
说明其他页面可能有内存泄漏现象
2、通过MAT查看 系统报出的疑似泄漏对象，通过incoming reference 查看引用链条
3、确定泄漏原因

1、Histogram具体对象泄漏
​ 1.1、按照包来找到最大的内存对象
​ 

并通过size来筛选其中大于1000在Retained Heap中输入，或者通过你所知道的可能占用内存的通用父类进行筛选在Package/Class中输入fragment

​

2、Dominator Tree分析具体图片对象泄漏
通过Dominator Tree来找到当前最占用内存的对象，从大到小进行排列
其中Shallow Heap 自身占用内存 和 Retained Heap引用对象内存，不成比例的，如红色标注的第一种情况，是很有可能是内存泄漏的重点
对于Shallow Heap 和 Retained Heap大小相等的这种对象，一般不是内存泄漏，但对象这么大本身也可能存在问题，可以查看具体对象是什么，再做优化

3、通过引用链具体分析内存引用点，查看drawable外部引用

并找到buffer，通过GIMP工具讲save的图片打开，注意宽高对应,save的格式为 xxx.data,

注意GIMP打开，调整图片宽高为 mBitmap对应的宽高，并设置格式为RGB Alpha，有时可能适当增加像素位移，然后查看对应的图片


之后通过GIMP工具查看到图片和本地app中图片对应，找到具体的图片泄漏代码，去除图片的引用，达到释放图片所占用内存的目的
4、通过GC找到对象泄漏点（注意去除虚引用，弱引用和软引用，具体app不便透露请见谅），来进行对象释放

注意红线方向从上到下是一条导致底部对象无法释放的引用链，其中任何一环的对象释放，都可以此环对象下的所有对象释放，那么我们可以看到，如果释放了mLocationListeners，那么底部的this对象便会释放，所以可以回到我们的代码进行将mLocationListeners引用释放

利用GPU Image直接渲染

如果我们configure传surface，我们可以通过gpu传递的方法，直接进行渲染，这样可以减少GPU <-> CPU之间的内存拷贝。首先configure时候传surface，我们调用ouputBuffer之后，获得输出缓存index，得到渲染时候，直接调用releaseOutputBuffer(handle,idx,true),则解码器的图像直接渲染到surface图像上了。

这样虽然效率高，但是弊端也很明显，第一，那就是图像后处理做不了。第二，这种方案依赖解码器缓存，这会带来一些问题。如果解码器被提前析构，则缓存内容都没有了。又或者一些播放业务逻辑对解码器缓存要求较多（比如倒放），这也做不了。
